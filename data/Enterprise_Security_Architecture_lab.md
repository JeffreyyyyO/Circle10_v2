# **Implementing Cloud Data Loss Prevention (DLP) using Amazon Macie**

## **Module 1: Environment Preparation, IAM Setup, and S3 Bucket Initialization**

### **1\. Lab Overview & Objectives**

In the modern digital landscape, protecting sensitive data—such as personally identifiable information (PII), credit card details, and confidential business documentation—is a critical challenge. This hands-on lab demonstrates how to implement a cloud-based **Data Loss Prevention (DLP)** strategy using **Amazon Macie**, an automated data security service that leverages machine learning and pattern matching to discover and protect sensitive data in Amazon S3.  
By the end of this module, you will be able to:

1. Update and prepare a security-focused guest OS environment (Kali Linux).  
2. Install and verify the AWS Command Line Interface (CLI).  
3. Provision an Identity and Access Management (IAM) user with custom security and administrative policies.  
4. Configure programmatic AWS access securely on your local terminal.  
5. Provision a globally unique Amazon S3 bucket via the CLI.

### **2\. Prerequisites**

* A virtualization platform (e.g., VirtualBox) hosting a security distribution such as Kali Linux.  
* Active administrative access to an AWS account.  
* Internet connectivity within the guest VM.

### **3\. Step-by-Step Instructions**

#### **Step 1: Prepare the Local Guest OS & Install AWS CLI**

We will perform our administration from the terminal of our Kali Linux virtual machine rather than relying exclusively on the AWS Management Console (GUI). This mimics real-world enterprise infrastructure deployment where automation and command-line execution are preferred.

1. Open your terminal in Kali Linux.  
2. Update the local package indexes to ensure you fetch the latest software versions:  
   `sudo apt update`

   *Enter your sudo password when prompted and wait for the process to complete.*  
3. Install the AWS CLI package using the Advanced Package Tool (APT):  
   `sudo apt install awscli -y`

   *Note: The \-y flag automatically accepts the installation prompts and agreements.*  
4. Verify that the AWS CLI is installed correctly by querying its version:  
   `aws --version`

   **Expected Output:**  
   An output showing the CLI version, the embedded Python version, and your operating system kernel type (e.g., aws-cli/2.x.x Python/3.13.x Linux/6.x.x...).

#### **Step 2: Configure AWS Identity and Access Management (IAM)**

To interact securely with AWS from an external terminal, we must configure a service account (IAM User) with restricted programmatic access.

1. Open your host machine's web browser, navigate to the AWS Management Console, and log in to your root or administrator account.  
2. Search for and select the **IAM** (Identity and Access Management) service.  
3. In the left navigation pane, select **Users** and click **Create user**.  
4. Configure the user properties:  
   * **User name:** outschool-user  
   * **Console access:** Checked (select *Autogenerate password* and ensure *Users must create a new password at next sign-in* is checked if they will also use the console, though this is optional for programmatic use).  
5. Click **Next**.  
6. On the permissions page, attach policies directly to grant the minimum necessary privileges for this lab:  
   * Click **Attach policies directly**.  
   * Search for and select **AmazonMacieFullAccess** (grants permissions to enable, configure, and manage Macie classification jobs).  
   * Search for and select **AmazonS3FullAccess** (grants full permission to create, write, read, and delete S3 resources).  
7. Click **Next**, review your configuration, and click **Create user**.  
8. **Critical:** Click **Download .csv** immediately to save the console login credentials and autogenerated password. This file contains sensitive access details that cannot be retrieved once you leave the screen.

##### **Generate Programmatic Access Keys**

1. From the IAM users list, select your newly created user: outschool-user.  
2. Navigate to the **Security credentials** tab.  
3. Scroll down to the **Access keys** section and click **Create access key**.  
4. Select the **Command Line Interface (CLI)** use case.  
5. Check the confirmation box acknowledging that you understand the best practices, then click **Next**.  
6. (Optional) Provide a descriptive tag, such as outschool-user-cli-access, and click **Create access key**.  
7. Click **Download .csv file** to save your Access Key ID and Secret Access Key. Keep this file highly secure.

#### **Step 3: Configure AWS CLI on Kali Linux**

With your programmatic credentials downloaded, you can securely authenticate your Kali Linux VM to communicate with AWS services.

1. In your Kali Linux terminal, initiate the configuration wizard:  
   `aws configure`

2. Populate the parameters prompted by the wizard using the details inside your downloaded access key CSV:  
   * **AWS Access Key ID:** Paste your Access Key ID.  
   * **AWS Secret Access Key:** Paste your Secret Access Key.  
   * **Default region name:** eu-west-1  
     *(Note: This specifies the EU West Ireland region. If you are operating from West Africa/Nigeria, eu-west-1 offers excellent latency, service coverage, and robust compliance features. You may also use us-east-1 or eu-central-1 depending on compliance needs).*  
   * **Default output format:** json  
     *(Note: JSON is highly structured, machine-readable, and ideal for troubleshooting and pipeline scripting).*  
3. Test your connection by listing any existing S3 buckets in your account:  
   `aws s3 ls`

   *If successful, this will run without error, returning a list of your buckets or a blank line if no buckets exist.*

#### **Step 4: Verify and Enable Amazon Macie via CLI**

Before scanning buckets, you must ensure Amazon Macie is active within your targeted region.

1. Check your existing Macie session properties:  
   `aws macie2 get-macie-session --region eu-west-1`

2. If Macie has never been activated, run the following command to enable it:  
   `aws macie2 enable-macie --status ENABLED --region eu-west-1`

3. Query the session details again to understand your deployment status:  
   `aws macie2 get-macie-session --region eu-west-1`

   **Key Configuration Metrics Explained:**  
   * createdAt: The precise UTC timestamp when Macie was initialized on your account.  
   * findingPublishingFrequency: Set to FIFTEEN\_MINUTES by default, dictating how frequently Macie updates findings.  
   * serviceRole: The AWS Service-Linked Role dynamically created to permit Macie to parse resources securely on your behalf.  
   * status: Displays ENABLED, indicating the engine is fully operational.

#### **Step 5: Provision a Globally Unique S3 Bucket**

Amazon Macie inspects files stored inside Amazon S3 buckets. We will provision a new bucket directly from our terminal.

1. Create a globally unique S3 bucket using the S3 API command. S3 bucket names are sharing a universal namespace, meaning your bucket name must be entirely unique across all AWS accounts:  
   `aws s3api create-bucket \`  
     `--bucket my-dlp-demo-bucket-123456 \`  
     `--region eu-west-1 \`  
     `--create-bucket-configuration LocationConstraint=eu-west-1`

   *(Ensure you replace 123456 with a personalized sequence of random numbers to guarantee uniqueness).*  
   **Why include LocationConstraint?** If you create an S3 bucket outside of the default us-east-1 (Northern Virginia) region, S3 requires you to explicitly state the constraint mapping matching your configured region. Omitting this parameter when targeting eu-west-1 will result in a validation error.  
   **Expected Output:**  
   `{`  
       `"Location": "[http://my-dlp-demo-bucket-123456.s3.amazonaws.com/](http://my-dlp-demo-bucket-123456.s3.amazonaws.com/)"`  
   `}`

   *This confirms the target S3 bucket has been successfully provisioned and registered in AWS.*

The core cloud infrastructure is set up, authenticated, and primed for DLP actions.

## **Module 2: Mock Data Creation, Macie Job Execution, Automated Remediation, and Resource Cleanup**

### **1\. Module Overview & Objectives**

Now that our core environment is authenticated and the target S3 bucket is provisioned, we will simulate a real-world data exposure scenario. We will craft a mock document containing highly sensitive PII and PHI (Protected Health Information), upload it, and trigger a surgical Macie discovery scan.  
To bridge the gap between detection and mitigation, we will write an automated event-driven **AWS Lambda** remediation handler in Python. This serverless script dynamically quarantines any files flagged with high-severity risks, moving them to a secure quarantine bucket and purging them from their public source. Finally, we will run safe environment teardown procedures to avoid unnecessary AWS subscription costs.  
By the end of this module, you will be able to:

1. Create mock datasets containing structured PII, PHI, and financial indicators using terminal-based text editors.  
2. Trigger, monitor, and troubleshoot CLI-based Amazon Macie classification jobs.  
3. Access and evaluate findings inside both the AWS Management Console and the CLI.  
4. Architect a serverless remediation pipeline using AWS Lambda to automatically quarantine compromised files.  
5. Execute cleanup commands to safely decommission active cloud infrastructure.

### **2\. Step-by-Step Instructions**

#### **Step 1: Create Mock Sensitive Data**

We will construct a mock data file locally. To prevent our directories from becoming cluttered, we will configure a dedicated workspace in the terminal.

1. In your Kali Linux terminal, navigate to your VM's Documents directory:   
   `cd ~/Documents`

2. Create a clean workspace directory named data and navigate inside:  
   `mkdir data && cd data`

3. Initialize an empty file named sensitive\_data.txt using the touch utility:  
   `touch sensitive_data.txt`

4. Open the file in the terminal-based text editor nano:  
   `nano sensitive_data.txt`

5. Populate the text file with simulated sensitive data. This file replicates accidental file exposure containing a combination of Personally Identifiable Information (PII), financial credentials, and Protected Health Information (PHI):  
   `Name: Victor Adebayo`  
   `NIN: 98765432109`  
   `SSN: 123-45-6789`  
   `Credit Card: 5647-7890-1234-0986`  
   `Health Status: Patient is critically ill.`  
   `Diagnosis: Terminal Respiratory Disease`  
   `Treatment Duration: 3 Months`

6. Save and exit the editor:  
   * Press Ctrl \+ X to initiate the exit sequence.  
   * Press Y to confirm saving the modified buffer.  
   * Press Enter to write the changes directly to sensitive\_data.txt.  
7. Verify the contents of your newly written file using the cat command:  
   `cat sensitive_data.txt`

#### **Step 2: Upload the Target Dataset to the S3 Bucket**

Using the configured AWS CLI, we will copy this sensitive dataset directly into our remote target storage container.

1. Execute the copy (cp) command to upload the file to your unique bucket:  
   `aws s3 cp sensitive_data.txt s3://my-dlp-demo-bucket-123456/sensitive_data.txt`

   *(Ensure you replace my-dlp-demo-bucket-123456 with your personalized bucket name configured in Module 1).*  
2. Verify that the file has successfully landed in your cloud object store:  
   `aws s3 ls s3://my-dlp-demo-bucket-123456/`

3. (Optional verification via AWS Console): Navigate to S3 inside your web browser, click on your bucket, and confirm sensitive\_data.txt is listed.

#### **Step 3: Trigger and Debug a Macie Classification Job**

We will now instruct Macie to evaluate our bucket. To execute this programmatically, we must first query our dynamic AWS Account ID to supply as an execution parameter.

1. Run the Security Token Service (STS) caller identity check to extract your 12-digit Account ID:  
   `aws sts get-caller-identity --query "Account" --output text`

   *Copy the returned numeric account string from your terminal.*  
2. Configure and run a one-time Macie classification job. This command maps our bucket definition, sets sampling criteria to 100%, and forces an immediate execution loop:  
   `aws macie2 create-classification-job \`  
     `--job-type ONE_TIME \`  
     `--name "my-dlp-demo-scan" \`  
     `--s3-job-definition '{"bucketDefinitions": [{"accountId": "YOUR_12_DIGIT_ACCOUNT_ID", "buckets": ["my-dlp-demo-bucket-123456"]}]}' \`  
     `--sampling-percentage 100 \`  
     `--description "Scan for sensitive data in my DLP demo bucket" \`  
     `--initial-run`

   *(Ensure you swap YOUR\_12\_DIGIT\_ACCOUNT\_ID with the output from the previous step and verify the targeted bucket name match exactly).*

##### **Note: Troubleshooting Case Sensitivity and Syntax**

During CLI job creation, Macie parsing parameters are strictly case-sensitive. Pay close attention to these common pitfalls:

* **Account Identifier Case:** The JSON key must be explicitly cased as accountId (using a lowercase a and capital I and D). Casing errors such as AccountID or account\_id will trigger validation exceptions.  
* **Buckets Key:** The list array mapping the targeted containers must be defined as "buckets": \["\<bucket-name\>"\]. Erroneously declaring it as "bulk" or "bucket" will block API ingestion.

##### **Expected Output:**

If entered correctly, the CLI will output a unique Macie job identifier and Amazon Resource Name (ARN):  
`{`  
    `"jobArn": "arn:aws:macie2:eu-west-1:123456789012:classification-job/a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",`  
    `"jobId": "a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d"`  
`}`

3. Query your job status to confirm execution is active or complete:  
   `aws macie2 list-classification-jobs --query "items[?name=='my-dlp-demo-scan'].jobStatus" --output text`

   *Wait until the status shifts from RUNNING or PAUSED to COMPLETE.*

#### **Step 4: Analyze Discovery Findings**

Once complete, Macie indexes discovery logs detailing matches against built-in managed data identifiers.

1. Query findings directly inside your terminal by executing:  
   `aws macie2 list-findings --max-results 10`

   *This output lists JSON blocks showing unique discovery IDs indicating where policy violations have occurred.*  
2. Navigate back to your web browser and load the **Amazon Macie Dashboard**.  
3. Under the left navigation tree, select **Findings**.  
4. Observe the incoming flags. You will see severity alerts mapped directly to your targeted bucket:  
   * **High Severity Find:** This alert notes the discovery of Personal Identifiable Information (PII) including Social Security Numbers (SSN), credit card configurations, and regional health data.  
   * **Object Detail Map:** Clicking on the high-severity finding details pane exposes the target asset name (sensitive\_data.txt), path variables, and the classification job ID responsible for logging the vulnerability.

#### **Step 5: Implement Automated Remediation via AWS Lambda**

To move beyond manual remediation, we can create a reactive serverless pipeline. Utilizing Amazon EventBridge, any High Severity Finding generated by Macie can trigger an AWS Lambda function designed to immediately move the exposed file into quarantine.

1. In the AWS Management Console, search for and navigate to **Lambda**.  
2. Click **Create function** and configure the properties:  
   * Select **Author from scratch**.  
   * **Function name:** my-dlp-remediation-handler  
   * **Runtime:** Python 3.12 (or current stable release)  
   * **Architecture:** x86\_64  
     *(Note: Standard x86\_64 is recommended here for maximum library compatibility. While arm64 offers up to 20% cost efficiency for execution workflows, x86\_64 remains the default choice to guarantee support across legacy environments and dependencies).*  
3. Under the permissions configuration dropdown, leave **Create a new role with basic Lambda permissions** selected. AWS will configure default CloudWatch logging permissions automatically.  
4. Click **Create function**.

##### **Implementing the Quarantine Code**

Inside your newly provisioned Lambda code dashboard, navigate to the code editor pane and replace any boilerplates with this functional python orchestration script. This script utilizes the boto3 AWS SDK to extract event logs, securely move (copy) the flagged file to a safe vault, and destroy the public original.  
`import json`  
`import boto3`

`# Initialize the AWS S3 client`  
`s3 = boto3.client('s3')`

`def lambda_handler(event, context):`  
    `"""`  
    `Automated Data Loss Prevention (DLP) Remediation Handler.`  
    `This function parses Macie finding events routed via EventBridge,`  
    `extracts the target S3 asset, copies it to a secure quarantine bucket,`  
    `and removes the compromised file from its source location.`  
    `"""`  
    `try:`  
        `# Extract finding properties from the event detail payload`  
        `finding = event.get('detail', {})`  
        `print("Processing Finding ID: %s" % finding.get('id', 'N/A'))`  
          
        `# Parse S3 resources affected by policy violations`  
        `resources_affected = finding.get('resourcesAffected', {})`  
        `s3_object = resources_affected.get('s3Object', {})`  
          
        `bucket_name = s3_object.get('bucketName')`  
        `object_key = s3_object.get('key')`  
          
        `if not bucket_name or not object_key:`  
            `print("No S3 resource identifiers located inside event payload. Halting execution.")`  
            `return {`  
                `'statusCode': 400,`  
                `'body': json.dumps('Incomplete S3 asset parameters parsed.')`  
            `}`  
              
        `# Define the target quarantined vault bucket`  
        `quarantine_bucket = "my-dlp-quarantine-bucket-123456"`  
        `copy_source = {`  
            `'Bucket': bucket_name,`  
            `'Key': object_key`  
        `}`  
          
        `print("Surgical isolation initiated. Copying asset %s from %s to %s..." % (object_key, bucket_name, quarantine_bucket))`  
          
        `# Execute cross-bucket object copy to secure the data`  
        `s3.copy_object(`  
            `CopySource=copy_source,`  
            `Bucket=quarantine_bucket,`  
            `Key=object_key`  
        `)`  
          
        `# Execute safe deletion of compromised asset from source public S3 bucket`  
        `print("Purging leaked public source file %s from bucket %s..." % (object_key, bucket_name))`  
        `s3.delete_object(`  
            `Bucket=bucket_name,`  
            `Key=object_key`  
        `)`  
          
        `log_message = "Successfully quarantined asset %s from source %s to storage vault %s." % (object_key, bucket_name, quarantine_bucket)`  
        `print(log_message)`  
          
        `return {`  
            `'statusCode': 200,`  
            `'body': json.dumps(log_message)`  
        `}`  
          
    `except Exception as e:`  
        `error_msg = "Error executing automation pipeline on S3 object: %s" % str(e)`  
        `print(error_msg)`  
        `raise e`

5. Click **Deploy** to compile the codebase and update your Lambda instance in the cloud.

#### **Step 6: Lab Cleanup and Asset Decommissioning**

Cleaning up lab resources is an essential practice to ensure you do not incur charges for idle assets.

1. In your Kali Linux terminal, run the file deletion command to purge sensitive\_data.txt from your public bucket:  
   `aws s3 rm s3://my-dlp-demo-bucket-123456/sensitive_data.txt`

2. Delete the primary S3 bucket via the S3 API command:  
   `aws s3api delete-bucket --bucket my-dlp-demo-bucket-123456 --region eu-west-1`

3. Verify that the bucket is deleted by refreshing the console view or listing S3 buckets in your terminal:  
   `aws s3 ls`

   *Your deleted bucket name should no longer be displayed.*

### **3\. Key Architectural Takeaways and Best Practices**

Implementing automated cloud-native data loss prevention aligns our operations directly with international and regional compliance standards (such as **GDPR**, **PCI-DSS**, and Nigeria's **NDPR**).  
To implement this robust security model at scale:

* **Centralize Logging:** Route all Amazon Macie alerts to a unified Security Information and Event Management (SIEM) tool—such as Splunk, or AWS Security Hub—to build correlated matrices across your infrastructure.  
* **Proactive Orchestration:** Implement automated EventBridge rule routing to trigger target lambdas dynamically, keeping sensitive files safe from public exposure before incidents escalate into compliance breaches.
